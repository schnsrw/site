Overview

Casual Editor is a browser-based document editor that feels like Word, not Google Docs — a ribbon, a real page-layout view, and a file-centric flow. Open a .docx from disk, edit it with WYSIWYG fidelity, save it back as .docx. When you want to co-edit, click Share, send a link, and everyone on it edits live.

The editor is a fork of eigenpal/docx-editor (MIT, React + ProseMirror, OOXML model preserved end-to-end). The real-time gateway is original Go — a stateless y-websocket relay with a pluggable host integration for storage.

Status — the fork is ahead of upstream on round-trip fidelity, header textboxes, VML preservation, multi-section preservation, and a few dozen smaller round-trip fixes. The Go backend just shipped M1 (in-memory inline host + WS broker + REST upload/download). Bundled Docker image lands with the first tagged release.

Features

Editing

  • Full WordprocessingML core — paragraphs, runs, tables, lists, sections, headers/footers, comments, tracked changes, hyperlinks, footnotes/endnotes, custom XML, math equations.
  • DrawingML rendering — pictures, modern shapes, textboxes (modern + VML fallback), wpg:wgp groups with per-child positioning, decorative v:rect / v:oval / v:line, connector lines, image hyperlinks.
  • Find & replace, multi-column lists, multi-select indent / outdent, theme-color resolution, tab leaders, paragraph borders (between + bar), per-section column counts.
  • File properties dialog wired to docProps/core.xml — author, subject, keywords, last-modified all round-trip.
  • Export to PDF via the browser's print pipeline (saved file defaults to <document>.pdf).

Collaboration

  • Real-time co-editing through a Yjs + y-prosemirror binding over the standard y-websocket protocol.
  • Share dialog uploads the active .docx to the gateway, mints a doc ID, returns a shareable link.
  • Connection-state UX — Joining session loading panel, persistent reconnecting/disconnected banner, per-peer avatars and remote cursors.

Round-trip

  • Per-tag round-trip audit on every CI run. Current baseline: 26 of 39 fixtures pristine; non-pristine fixtures only drop in a few non-VML edge cases now that the enricher captures the source envelope XML for verbatim replay.
  • Three-way fidelity comparison harness in CI — us vs LibreOffice (soffice --convert-to docx) vs OnlyOffice DocumentBuilder — runs on push to main. Fails the workflow if our pristine share drops below the floor.

Architecture

Three layers, with the backend deliberately stateless. Document persistence is delegated to a host integration (inline / WOPI / JWT-API) so the gateway slots into existing storage stacks rather than carrying its own. 

Browser
  ├─ <DocxEditor/> — React + ProseMirror, OOXML model preserved
  ├─ y-prosemirror ySyncPlugin
  └─ Y.Doc ⇄ y-websocket transport
                  │
                  ▼
Go gateway (stateless)
  ├─ WS broker — fans every binary frame to peers in the same room
  ├─ In-memory Y.Doc per live session (dropped on last disconnect)
  ├─ host.Integration interface
  │    ├─ inline   — in-process map (the v0 share-link flow)
  │    ├─ wopi     — full WOPI HTTP (future)
  │    └─ jwtapi   — JWT-secured REST (future)
  └─ Snapshot worker → host on room drain (Y.Doc → .docx)
                  │
                  ▼
Storage host (external)
  GetFile to seed a fresh session
  PutFile to persist on drain

Stateless invariant. No database, no on-disk update log. The only live state is the in-memory Y.Doc while a session is active; when the last client disconnects it's gone, and the next session re-seeds from the host.

Setup

Try the hosted demo

doc.schnsrw.live is the single-user demo (no backend behind it). Drag a .docx on, edit, save back to disk.

Self-host with Docker

The bundled image carries the editor SPA and the Go gateway, so co-editing works out of the box on the host you run it on. 

# Clone + build + run locally
git clone https://github.com/schnsrw/docx
cd docx
docker compose up
# open http://localhost:8080
# upload a .docx, click Share, send the link

# Dev mode (Vite hot reload + Go run)
docker compose --profile dev up

After the first tagged release, docker run -p 8080:8080 schnsrw/casual-editor:latest will pull the pre-built image from Docker Hub.

Develop on the editor directly

cd docx/docx-editor
bun install
bun run dev           # Vite at :5173
bun run typecheck
bun test
bun run test:e2e

Develop on the gateway directly

cd docx/backend
go vet ./...
go test -race ./...
go run ./cmd/gateway   # listens on :8080

Configuration

All configuration is environment variables — no config files. The bundled image picks safe defaults; override only what you need.

Gateway (Go)

GATEWAY_ADDR
TCP address to bind to. Default: :8080.
STATIC_DIR
Directory to serve the editor SPA from. When set, the gateway also handles / (with an index.html SPA fallback) so the editor and API share one origin. The bundled image sets this to /srv/static. Unset in local dev where Vite serves the SPA on :5173.

Editor build (Vite)

VITE_BACKEND
HTTP base for the gateway. Used by the share dialog's upload and the seed download in collab sessions. Unset in production builds → falls back to window.location.origin (the bundled-image story). Set this in dev when Vite is on :5173 and the gateway on :8080.
VITE_COLLAB_ENABLED
true to expose the Share button. The Pages demo builds with this off (no backend behind the demo origin); the Docker image sets it to true.

Fidelity comparison CI

FIDELITY_FLOOR
Pristine-share floor for the three-way comparison harness (0–1). The workflow fails when our pristine share falls below this. Default: 0.5. Raised as we close gap-matrix items.
LIBREOFFICE_BIN
Path to soffice if it's not on PATH. The harness probes a few defaults.
ONLYOFFICE_DOCBUILDER_IMAGE
Override the Docker image tag for OnlyOffice DocumentBuilder. Default: onlyoffice/documentbuilder:latest.

Fidelity benchmark

The audit script tallies every OOXML element name in the input document.xml against the re-serialized output; tags whose count drops are silent round-trip losses.

EngineApproachPristine on our corpus
Casual Editor Parse → model → serialize, with raw-XML envelope capture for drawings 26 / 39 (≈ 67%); target ≥ 90% before desktop ship
LibreOffice Online / Collabora Server-side C++ load + re-emit; mature, 25+ years Industry-reported ~96–98% on representative business docs (no direct benchmark on our corpus yet — wired into the CI harness)
OnlyOffice DocumentBuilder Server-side C++ x2t converter; ~10 yrs Same band as LibreOffice on representative docs

The point isn't to clear OnlyOffice's bar — those are mature server-side renderers with years of OOXML coverage. The point is to clear the bar for a lightweight alternative: a couple of megabytes of JS + a small Go binary instead of a 1+ GB native suite, while staying fidelity-honest on the docs people actually edit.

FAQ

Why a fork of eigenpal/docx-editor instead of building from scratch?
The OOXML model and round-trip pipeline are weeks of work to design well. eigenpal does both. Forking lets us send fidelity fixes back upstream while building the gateway + bundled distribution on top.
Why Go for the gateway and not Node or Rust?
Go's standard library is enough to speak the y-websocket protocol cleanly, the runtime is small (no Node-style memory tax for an idle session), and the binary statically links. The full reasoning lives in docs/05-backend-design.md.
Why no database?
The gateway is the collaboration layer, not the storage layer. Persistence is delegated through host.Integration — inline for the v0 share-link flow, WOPI / JWT-API for integration with existing document hosts. Keeps the gateway disposable.
Will Casual Editor support .odt, .md, .txt?
Planned, post-docx-fidelity-floor. Editing stays docx-only — other formats land via input parsers + output converters that bookend the canonical model. .doc is explicitly out of scope.
Is co-editing in the Pages demo?
No. Co-editing requires the gateway, and there's no backend behind doc.schnsrw.live. The bundled Docker image is the collab distribution. Tauri desktop is single-user only.
What's "pristine" mean in the round-trip audit?
Every OOXML element name in the input document.xml appears at least as many times in the re-serialized output. Tag-count parity, not byte-level — that's a stricter bar (semantically-equivalent XML can fail it) but a useful one for catching silent drops.

Roadmap

  1. Closer to 90% pristine. Close the remaining non-VML round-trip drops (w:fldChar in complex fields, edge-case w:rPr / paragraph-properties ordering, multi-section w:pgNumType). The audit surfaces them per-fixture.
  2. Y.Doc → .docx serializer worker pool (M2). Right now the gateway re-serves the original upload on drain. The next milestone runs a Bun worker pool that turns the live CRDT state into a fresh .docx before snapshot.
  3. JWT host integration. Authenticated fetch / snapshot against an external host, configured via env. WOPI follows.
  4. Web Workers for parse / serialize. Move the heavy paths off the main thread so 100+ page docs stop stuttering the UI.
  5. Floating-image text wrap. The biggest remaining visual gap on the layout side.
  6. .odt + .md + .txt converters. Lossy input / output converters around the canonical docx model. .doc stays out of scope.
