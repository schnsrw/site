Overview

Casual Desktop packages the Casual Editor and Casual Sheets web cores as a native binary via Tauri. Your files stay on your machine; there is no cloud round-trip and no collab server. A small download, a native window, native file dialogs.

Status — early scaffolding. The first binary lands after the web fidelity hits its 90% pristine floor; building on a moving target wastes cycles.

Approach

Casual Desktop is intentionally not a fresh editor. The web codebases — React + ProseMirror for documents, Univer for sheets — already solve the hard problems: rendering, formulas, OOXML round-trip, paste, undo. The desktop shell adds three thin layers:

  1. OS file dialogs. Open / Save / Recent driven by Tauri's native file APIs instead of <input type=file>.
  2. Local file persistence. The web cores hand bytes; the Tauri Rust backend writes them to the path the user picked.
  3. Optional native parsing. When a doc is big enough for the in-browser parser to feel sluggish, route through a Rust crate on the Tauri side and hand the model into the React view. Pure perf optimization, not fidelity.

No collab. Co-editing is the Docker image's job, not the desktop binary's.

Roadmap

  1. Scaffold. Tauri app shell + Casual Editor or Casual Sheets web build packaged as the WebView content.
  2. Native file flow. Open / Save / Save As wired through Tauri's dialog + fs APIs.
  3. Recent files. Per-user JSON store, pinned-files surface in the editor's File menu.
  4. Multi-platform CI. Tauri release pipeline for macOS (universal), Linux (AppImage), Windows (MSI).
  5. Native parser handoff. Rust crate consumed from both the Tauri backend (native) and the web build (via WASM) for big-doc perf.

FAQ

Why Tauri and not Electron?
The binary is an order of magnitude smaller, the runtime uses less RAM, and there's no Chromium bundled — the OS WebView handles rendering. The trade-off is platform-specific WebView quirks; for an office-style editor those are well-charted.
Will the desktop app open .docx and .xlsx from one binary?
That's the goal. Probably ship two binaries first (one per editor), merge into a single Casual Office app once the file-type routing is clean.
Will it support co-editing?
No. The collab story is Docker. The desktop binary is for the offline / single-user case where collab is a liability (latency, server dependency, data leaving the machine).
License?
To be decided at launch. The web cores stay under their existing licenses (MIT for Casual Editor, Apache-2.0 for Casual Sheets). The desktop shell will likely match.
